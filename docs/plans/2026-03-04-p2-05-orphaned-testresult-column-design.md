# P2-05 Orphaned TestResult Column Removal Design

## Background

Phase 2 issue P2-05 reports a schema drift in `TestResults`: migration `20260204080821_StagingReadyMigration` added an orphaned column `TestResult_Measurement_ParameterId` with its own FK and index. EF still maps a shadow property to that column via `TestResult.Parameter` navigation, so EF sees the column as part of the model and will not drop it.

Constraint: no manual edits to migration SQL or migration C# code. All schema changes must be generated by EF migrations.

## Goals

- Remove the orphaned `TestResult_Measurement_ParameterId` column, FK, and index via a generated EF migration.
- Align the EF model with the intended schema by removing the shadow FK mapping.
- Keep service behavior and DTO output consistent after model changes.

## Non-Goals

- No manual SQL in migrations.
- No changes to the core `Measurement` ownership or schema besides removing the orphaned column.

## Options Considered

1. Remove the `TestResult.Parameter` navigation and FK mapping so EF no longer needs the shadow column, then generate a migration to drop it. (Recommended)
   - Pros: pure EF migration, aligns model to intended schema, clean drop.
   - Cons: requires service/test updates where the navigation was used.
2. Keep navigation and remap it to `Measurement_ParameterId` explicitly.
   - Pros: preserves navigation.
   - Cons: higher configuration complexity, may still leave the orphaned column without manual SQL.
3. Manual SQL in a migration to drop the column/FK.
   - Pros: direct and quick.
   - Cons: violates the no-manual-migration rule.

## Decision

Proceed with option 1: remove the `TestResult.Parameter` navigation + FK mapping, update services/tests to use `Measurement.ParameterId`, and let EF generate the drop migration.

## Design

### Model and Mapping

- Remove the `Parameter` navigation from `TestResult`.
- Remove the `.HasOne(e => e.Parameter)` mapping from `TestResultConfiguration` so EF stops creating the shadow FK property `Measurement_ParameterId` mapped to `TestResult_Measurement_ParameterId`.

### Data Flow and Services

- Replace any usage of `TestResult.Parameter` with explicit lookup via `Measurement.ParameterId`.
- Update `TestResultService` queries so parameter name/value lookups use the measurement FK directly.
- Update integration tests that relied on `TestResult.Parameter` navigation to use the revised lookup path.

### Migration

- After model changes, run `dotnet ef migrations add DropOrphanedTestResultMeasurementParameterIdColumn`.
- Verify the generated migration drops the `TestResult_Measurement_ParameterId` column, FK, and index.
- No manual edits to migration files.

### Tests

- Update `TestResultServiceIntegrationTests` as needed to reflect the new data access path.
- Run the standard test suite (`dotnet test backend/ -q`) after the migration is generated.

## Affected Files (Expected)

- `shared/Models/TestResult.cs`
- `backend/src/Quater.Backend.Data/Configurations/TestResultConfiguration.cs`
- `backend/src/Quater.Backend.Services/TestResultService.cs`
- `backend/tests/Quater.Backend.Core.Tests/Services/TestResultServiceIntegrationTests.cs`
- `backend/src/Quater.Backend.Data/Migrations/*DropOrphanedTestResultMeasurementParameterIdColumn*.cs`

## Risks and Mitigations

- Risk: EF still does not generate a drop migration if any leftover mapping remains.
  - Mitigation: re-check `QuaterDbContextModelSnapshot.cs` for any remaining mapping to `TestResult_Measurement_ParameterId` and remove it via model changes only.

## Success Criteria

- EF generates a migration that drops the orphaned column and related FK/index.
- TestResult behavior and tests continue to pass with the updated lookup path.
